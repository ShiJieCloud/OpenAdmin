# 项目 Git 提交信息规范
## 一、规范目的
1. 统一提交日志风格，清晰区分**功能、修复、优化、文档**等变更类型
2. 快速生成版本更新日志、自动化CHANGELOG
3. 方便代码回溯、代码评审、分支合并与问题定位
4. 适配开源社区协作习惯，降低多人协作沟通成本

## 二、基础提交格式（强制统一）
### 标准结构
```
<类型>[可选作用域]: <简短描述>

[可选详细正文说明]

[可选关联标签/关闭issue]
```
### 格式约束
1. **第一行**：标题行，**不超过50字符**，结尾无标点
2. **空一行**：标题与正文必须换行分隔
3. **正文行**：每行不超过72字符，描述变更原因、业务逻辑、实现思路
4. **尾部**：可关联 issue、任务ID、关闭需求

## 三、提交类型枚举（固定不可新增）
| 类型 | 英文标识 | 使用场景 |
| ---- | ---- | ---- |
| 新增功能 | feat | 新增业务功能、接口、组件、模块 |
| 缺陷修复 | fix | 修复线上bug、逻辑错误、异常报错 |
| 代码优化 | refactor | 重构代码、优化结构、不改变业务逻辑 |
| 样式调整 | style | 格式化代码、缩进、空格、注释排版，无逻辑变更 |
| 文档更新 | docs | 修改README、注释、接口文档、使用教程 |
| 性能优化 | perf | 提升运行速度、减少内存占用、优化查询效率 |
| 测试相关 | test | 新增单元测试、集成测试、修复测试用例 |
| 构建配置 | build | 修改打包脚本、依赖版本、编译配置、CI/CD |
| 工程配置 | chore | 调整项目目录、忽略文件、工具配置、无关业务修改 |
| 回滚撤销 | revert | 撤销某次历史提交 |

## 四、作用域规范（可选，建议大型项目必写）
作用域用于标注**修改模块/文件/业务域**，统一小写
### 常用作用域示例
- `api`：接口层
- `db`：数据库、表结构、SQL
- `auth`：权限、登录、认证
- `cache`：缓存、Redis相关
- `log`：日志模块
- `ui`：前端页面、组件
- `config`：全局配置
- `utils`：工具类、公共方法
- `entity`：实体类、数据模型
- `task`：定时任务、异步任务

### 带作用域示例
```
feat[auth]: 新增手机号登录接口
fix[db]: 修复分页查询偏移量错误
refactor[utils]: 统一日期格式化工具方法
```

## 五、正文编写规则
1. 说明**为什么改**，而非只写改了什么
2. 写明影响范围、兼容情况、注意事项
3. 涉及 Breaking Change 必须明确标注
4. 多改动点使用 `-` 无序列表分行

### 正文示例
```
feat[user]: 新增用户实名认证功能

1. 接入实名信息核验接口
2. 新增实名状态数据库字段
3. 限制未实名用户部分功能访问

关联 #123
```

## 六、Issue 关联语法（开源必备）
- 关联任务：`#数字`
- 关闭issue：`fix #100` / `close #100` / `resolve #100`
- 批量关闭：`close #100, #101`

示例：
```
fix[api]: 修复文件上传大小限制异常
close #89
```

## 七、禁止违规提交格式
1. 纯无意义提交：`更新代码`、`改一下`、`修复bug`
2. 中英文混杂乱写、首字母不统一
3. 标题过长堆砌大量内容
4. 一次提交混杂多种无关类型修改
5. 提交信息全大写/全拼音/网络用语

## 八、标准提交示例（直接套用）
### 1. 新增功能
```
feat[cache]: 实现分布式锁基础封装
支持超时自动释放，适配业务并发场景
```
### 2. Bug 修复
```
fix[task]: 修复定时任务重复执行问题
调整任务锁过期时间，避免集群多实例抢占异常
```
### 3. 代码重构
```
refactor[entity]: 统一实体类基础字段结构
抽离公共父类，减少重复代码
```
### 4. 文档修改
```
docs: 更新项目快速部署教程
补充Mac M系列芯片环境搭建步骤
```
### 5. 依赖与工程调整
```
chore: 升级项目依赖版本，清理无用依赖
```
### 6. 性能优化
```
perf[db]: 优化首页列表查询SQL
增加联合索引，减少全表扫描耗时
```

## 九、分支提交配套约定
1. **feature/xxx** 功能分支：统一使用 `feat` 类型
2. **hotfix/xxx** 紧急修复分支：统一使用 `fix` 类型
3. **release/版本号** 发布分支：以`build、docs、chore`为主
4. **dev** 开发分支：禁止随意合并大体积提交，保持粒度细小

## 十、简易提交模板（可直接配置 Git）
```txt
# 提交类型: feat/fix/refactor/docs/style/perf/test/build/chore/revert
# 作用域(选填): api db auth cache log config utils
# 第一行: 简短描述(≤50字)
# 空一行
# 详细说明修改内容、原因、影响
# 最后关联 issue: close #xxx
```
